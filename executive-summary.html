Executive summary — monorepo protection, asset storage, CI compute

Three decisions an engineering manager needs to make before migrating conjure into a per-path-protected monorepo: which protection rules actually exist on GitHub vs. which the plan assumed; whether to use Git LFS or Cloudflare R2 for binary assets; and whether to stay on GitHub-hosted runners or move to a managed alternative.

TL;DR 1. Protection findings 2. LFS vs R2 3. CI runners Combined recommendations

TL;DR — three decisions

1. Protection model

One ruleset, required_approving_review_count = 0 + require_code_owner_review = true, plus path-scoped required status checks that always run and skip-as-success. Empirically delivers US-4, US-5, US-6, US-7, and US-8 (in the form “author merges when sole codeowner of touched paths”). US-3 dropped — rulesets aren’t path-scoped, all paths go through PRs.

2. Binary assets

For conjure’s pattern (already running R2 for conjure-artifacts): R2 wins by >20× on bandwidth-heavy workloads. LFS only beats R2 when assets stay tiny (<5 GB) and CI bandwidth is negligible. Keep LFS for build inputs versioned with code; route everything else through R2.

3. CI runners

Blacksmith for fastest activation; RunsOn (in your own AWS) if a senior SRE owns CI and compliance matters. Skip BuildJet (shutting down 2026‑03‑31), skip self-hosted Hetzner unless usage is heavy and predictable.

1. Per-path protection — the final shape

The validation exercised all 20 user stories. 15 pass under the recommended ruleset; US-3 dropped as out-of-scope; two need a live claude session to verify; US-20 is workflow-logic verified but needs a bot identity to live-fire. The earlier “US-3 / US-4 / US-5 / US-8 are all plan-vs-platform mismatches” reading was wrong — an artifact of testing with the PR author as the only CODEOWNER.

The recommended config

name: main-protection
target: branch
bypass_actors: []
rules:
  - deletion
  - non_fast_forward
  - required_linear_history
  - required_status_checks: [ci-gate, ops-gate, pass, <more path-scoped checks>]
  - pull_request:
      required_approving_review_count: 0
      require_code_owner_review: true
      require_last_push_approval: false
      allowed_merge_methods: [squash, rebase]

Why count = 0 + codeowner = true: with count ≥ 1, every PR (spike included) needs a review → US-5 breaks. With count = 0, GitHub gates only on the codeowner rule, which fires only when codeowner-matched paths are in the diff. Spike-only PRs sail through. Prod-touching PRs need a peer codeowner. PRs where the author is the only codeowner of all touched paths merge through silently because no valid reviewer exists — that’s how the author-merges-own-prod-PR pattern works without literal self-approval.

Three things still don’t fit natively (and that’s OK)

  1. US-3 (direct push to main for one path tier) — rulesets are per-branch. Dropped from scope; ~30s per spike PR is fine.
  2. US-8 as literally worded (“author self-approves”) — GitHub blocks addPullRequestReview with “Can not approve your own pull request”. The intent (author can ship their own prod change) is preserved by the sole-codeowner mechanism above.
  3. UI-only merge enforcement — GitHub’s web UI and gh pr merge hit the same endpoint. No native way to allow one and forbid the other. Treat as policy.

CI workflow gotcha (still relevant)

Top-level on: pull_request: paths: [prod/**] makes the workflow not trigger when only non-matching paths change — the required check then shows “expected / missing” forever and PRs stick. Fix: the workflow always starts; the path filter is an inner job (dorny/paths-filter); the gate job uses if: always() to report skip-as-success.

2. Git LFS vs Cloudflare R2 — pricing & fit

Conjure already runs R2 (apps/api/wrangler.toml binds ATTACHMENTS_BUCKET = "conjure-artifacts" on the Workers API). Secrets for OBJECT_STORE_ACCESS_KEY / SECRET_KEY / ENDPOINT are catalogued in SECRETS_MAP.md. The infrastructure to use R2 for repo assets too is already in place.

Operational tradeoffs

LFS pros

  • Zero new tooling — git lfs install + .gitattributes and it just works.
  • Smudge filter pulls blobs transparently. CI just sets actions/checkout with lfs: true.
  • One ACL surface: GitHub repo permissions cover assets too.
  • Versioned alongside code. Reverting a commit also reverts the asset.

LFS gotchas

  • Bandwidth metering is per-clone, per-fork, and per-CI-run. Public-fork pulls bill you.
  • No per-team quota visibility — you only see the org-wide meter.
  • Rewriting LFS history (e.g. after an accidental commit) is painful and bandwidth-charged.
  • Fork-friendliness: opening to public OSS contributions can spike the bill arbitrarily.

R2 pros

  • Zero egress — bandwidth-heavy workloads are essentially free.
  • S3-compatible API — works with existing tooling (rclone, awscli with endpoint URL, etc.).
  • Infrastructure already paid for in conjure's Workers stack.
  • Auth via signed URLs, per-bucket; multiple buckets cost nothing.

R2 gotchas

  • Not git-native — you lose “the commit and the asset travel together.” Compensate with a manifest file in git pinning asset hashes.
  • Custom transfer agent for LFS-via-R2 exists (git-lfs-transfer-agent-s3 and similar) but is community-maintained, not vendor-blessed.
  • Class A ops (writes) priced higher than B — a CI pipeline that uploads many small files can rack ops charges.
  • Different ACL surface from GitHub; rotation of OBJECT_STORE_* keys is your job.

Recommended split for conjure

  1. Keep Git LFS for the small, build-required, code-versioned set: design tokens, font files, fixture binaries used by tests, anything where commit == asset. Budget: stay under one $5/mo data pack — in practice that means < 50 GB both ways.
  2. Use R2 for everything else: training data, large media, snapshot dumps, ML weights. Reference them by content hash in a manifest under data/manifest.json with a small fetcher script. Reproducibility preserved without bloating LFS.
  3. Public OSS-readiness check: if the repo will ever go public, LFS pulls from forks land on your meter. R2 with signed URLs caps that risk.

The three options actually worth shortlisting

Blacksmith — default pick

Why: tier-1 benchmarks (fastest x64 CPU + fastest cache in independent testing), one-label drop-in, 3,000 free min/mo, mature customer base (1,200+ cited).

Math for conjure: 10-person team @ 20k min/mo ≈ $68/mo after free tier vs $120 on GitHub 2-vCPU. ~2× wall-clock improvement is the real win, not the dollar delta.

Downsides: code runs in their cloud (same posture as GitHub-hosted, no compliance gain), static IPs cost $100/mo each, macOS pricing is unfriendly.

RunsOn — if you have SRE bandwidth

Why: €300/yr flat license, 4-vCPU runs at ~$0.0012/min on EC2 (no markup), code never leaves your AWS account — SOC-2/HIPAA-friendly. Tier-1 benchmarks identical to Blacksmith.

Math: ~50% cheaper than Blacksmith per minute. Spot instances add another 50–70%.

Downsides: AWS-only. CloudFormation install. You own the AWS bill and IAM — non-zero SRE work for upgrades, capacity, and audit.

Depot — for Docker-heavy steps

Why: remote Docker builders with persistent NVMe cache eliminate the slowest part of most CI pipelines. Image builds drop 5–10×.

Math: raw per-minute rate ($0.008) is the same as GitHub legacy — savings come from the cache making jobs short.

Downsides: highest lock-in (proprietary BuildKit fork, their cache format). ToS forbids public benchmarking — you cannot verify their claims pre-purchase. Compose with Blacksmith for the rest of the pipeline.

When the math does not work

  • < 1,000 CI min/mo: stay on GitHub-hosted. Setup tax beats savings.
  • Public OSS w/o sponsor budget: GitHub-hosted is free on public; alternatives generally aren't. Some have OSS programs — apply.
  • Mostly macOS/Windows: Blacksmith macOS is more expensive than GitHub. Stay put.
  • Bursty < 5k min/mo: RunsOn / Actuated flat fees become per-minute-expensive. Stay pay-as-you-go.
  • Strict no-third-party-execution policy: only RunsOn (your AWS), Actuated (your metal), or self-hosted qualify.

Risk callouts

  1. Vendor longevity is real. BuildJet was a top-3 player a year ago; it goes dark March 31, 2026. Pick someone with scale (Blacksmith claims 12× YoY revenue growth) or an OSS exit ramp (Ubicloud, Actuated).
  2. Cache speed often matters more than CPU. A pnpm + Turborepo pipeline spends 25–40% of wall-clock on cache save/restore. A 5× cache speedup beats a 2× CPU bump in that profile. The Turborepo / Bazel integrations Namespace ships are worth a serious look.
  3. “No public benchmarks” clauses are a tell. Depot's ToS forbids them — negotiate a trial; do not buy on faith.
  4. Self-hosted runners are a documented backdoor vector (Praetorian, Sysdig). Ephemeral-per-job is mandatory if you go this route.
  5. Billing surprises: Blacksmith static IPs ($100/mo each), Depot cache overage ($0.20/GB-mo), Hetzner hourly minimum. Set monthly spend alerts before adopting.

Combined recommendation for conjure

  1. Protection: ship the ruleset with required_approving_review_count: 0 + require_code_owner_review: true. Spike-only PRs sail through with no review (codeowner rule has nothing to fire on); prod-touching PRs need a peer codeowner; PRs where the author is the only codeowner of all touched paths merge silently (US-8 intent). Drop US-3 from scope (no path-scoped direct push); reword US-8 to “author can merge own PR” not “author can self-approve” (GitHub blocks the latter at the API).
  2. Assets: route net-new binary content (training data, ML weights, large media) through the existing conjure-artifacts R2 bucket — pin by hash in a tracked manifest. Keep Git LFS only for asset categories where commit-coupled history genuinely matters and total volume stays < 1 data pack.
  3. Compute: migrate to Blacksmith as the default runner (one-line label swap). Cost +speed both improve; activation energy is one PR. If/when CI bill exceeds ~$200/mo or compliance becomes a concern, evaluate RunsOn-in-our-AWS as the next hop. Add Depot for the Docker step only if image builds are >30% of pipeline wall-time.
