The decided shape (read this first)

One ruleset main-protection: restrict deletions, require linear history, block force-push, require PR-with-codeowner-review, required status checks ci-gate / ops-gate / pass. No bypass actors.

One PR rule: required_approving_review_count = 0 + require_code_owner_review = true.

One workflow shape: every required check starts always, uses an inner dorny/paths-filter job to decide whether to do work or skip-as-success.

The user-story matrix below holds against that exact config — verified by opening real PRs and observing outcomes.

What the experiment proved

I temporarily switched CODEOWNERS to @octocat (so codeowner ≠ PR author) under the recommended ruleset, opened two fresh PRs, and observed:

PRDiff touchesCodeowner match?Outcome
#6 chore/codeowners-test .github/CODEOWNERS Yes (.github/** @octocat) blocked on merge — “the base branch policy prohibits the merge”
#7 chore/codeowners-author-test spikes/mock-spike/README.md No merged with zero reviews

Then I reset CODEOWNERS back to @ronaldluc (author = sole codeowner) and opened PR #9 touching prod/mock-app/frontend/index.js. gh pr merge 9 --squash succeeded with no flags — GitHub silently no-ops the codeowner rule when no valid reviewer exists other than the author.

Decisions about the original plan

US-3 — dropped

The plan wanted spike-only changes to push directly to main. GitHub rulesets aren’t path-scoped; “Require a pull request before merging” is binary across the branch. Building a workflow gate to fake this isn’t worth the ~30 seconds saved per spike PR. All paths now go through PRs.

US-8 — wording fixed

The plan’s “author can self-approve their own prod PR” is literally impossible — GitHub blocks the API call with “Can not approve your own pull request.” The intent (the author can ship their own prod change once CI is green) is preserved by the count=0 + codeowner=true pattern: when author = sole CODEOWNER of all touched paths, the rule no-ops; otherwise a peer codeowner approves. No self-approval is ever attempted.

“Allow merge only from the UI, not CLI” — not possible

The GitHub web UI and gh pr merge hit the same REST endpoint. There is no native way to allow the merge action through one and not the other. Three practical alternatives:

  1. Policy: don’t run gh pr merge from automation. Every merge is audit-logged with the client identifier.
  2. Token scope: PATs scoped to Contents: read can’t merge; only the interactive web session (cookie auth) can.
  3. Bypass actors do not help: bypass_mode: pull_request still allows CLI bypass too — empirically verified on PR #8.

Recommended config to ship (the same one currently live)

name: main-protection
target: branch
bypass_actors: []
conditions: { ref_name: { include: ["~DEFAULT_BRANCH"] } }
rules:
  - deletion
  - non_fast_forward
  - required_linear_history
  - required_status_checks:
      strict_required_status_checks_policy: false
      required_status_checks:
        - ci-gate           # path-scoped, skip-as-success
        - ops-gate          # path-scoped, skip-as-success
        - pass              # pre-commit-fast (always runs)
  - pull_request:
      required_approving_review_count: 0
      require_code_owner_review: true
      dismiss_stale_reviews_on_push: false
      require_last_push_approval: false
      allowed_merge_methods: [squash, rebase]

Plus: every required status check is implemented as a workflow that always starts and uses an inner changes job (or dorny/paths-filter) to decide between real work and skip-as-success. Add new path-scoped checks (e.g. ai-review, ai-visual-review) to required_status_checks the same way.

Lessons for the conjure migration

  1. US-3 is gone from scope. All paths go through PRs.
  2. US-8 doesn’t require self-approval. Use count=0 + codeowner=true; with the author = sole codeowner of touched paths, GitHub no-ops the rule and the merge button works. With multi-codeowner teams (the real conjure case), peers approve normally.
  3. US-5 works natively in the same config — the codeowner rule simply has nothing to fire on for spike-only PRs.
  4. Internal paths-filter, not top-level paths:. Required checks plus top-level path filters = stuck PRs. Always run the workflow, gate inside.
  5. UI-only enforcement isn’t a GitHub feature. Treat it as policy.
  6. Dependabot SLA is minutes. Tighten the migration timeline.
  7. US-20 Step B (live bot test) stays open. Schedule it for migration day.
