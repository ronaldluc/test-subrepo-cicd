The second pass confirmed all the previously-blocked positive stories (US-1, US-2, US-4, US-6, US-7 under enforcement). It also surfaced three places where the original plan is incompatible with how GitHub's protection model actually works. Those are not bugs in the configuration — they're design choices the team needs to make before migrating conjure.

Three plan-vs-GitHub mismatches

US-8 — self-approval is not a configurable thing

The plan says: “Author can self-approve their own prod PR if they are a CODEOWNER (GitHub allows this if branch protection permits).”

GitHub does not allow this. The GraphQL API returns this error when the PR author tries to approve their own PR, no matter what the ruleset says:

failed to create review: GraphQL: Review
— Can not approve your own pull request

Implication for the migration: a single-CODEOWNER tier (today: prod/** → @ronaldluc alone) means no one can ship prod changes without either (a) a second collaborator gaining code-owner status, (b) bypass-actor exemptions in the ruleset, or (c) moving review enforcement out of GitHub's built-in system entirely.

US-3 + US-5 vs US-4 — rulesets aren't path-scoped

The plan wants three behaviors simultaneously:

  • US-3: direct push to main with a spike-only change succeeds
  • US-5: spike-only PR doesn't need review
  • US-4: prod PR requires CODEOWNER review

Branch rulesets in GitHub apply to a branch, not to file paths within it. With a single ruleset on main:

  • required_approving_review_count = 0 + require_code_owner_review = true: PR #3 (prod-only) merged with zero reviews. CODEOWNERS isn't enforced when the count is 0.  US-4 fails
  • required_approving_review_count = 1 + require_code_owner_review = true: PR #2 (spike-only) shows REVIEW_REQUIRED. Spike changes now need an approval.  US-5 fails
  • Any ruleset that requires a PR for merge to main blocks direct push regardless of path.  US-3 fails

Resolution paths:

  1. Accept the tradeoff — ship with count=1 + codeowner=true and let spike PRs need a review too. Simplest.
  2. Add a workflow gate — a check named e.g. codeowner-required that runs only when CODEOWNERS-matched paths are touched, fails without a code-owner approval, and is itself a required status check. Gives path-scoped review enforcement.
  3. Drop US-3 — in practice "direct pushes to main" are rarely worth keeping. Make spike work happen on branches like everything else.

CI required-checks + top-level paths: filter (the silent footgun)

Not a plan error — a hidden requirement. When a workflow uses on: pull_request: paths: [prod/**] at the top level and a PR touches only spike files, the workflow never starts. Required status checks that reference jobs in that workflow show "expected / missing" indefinitely and the PR is stuck.

The plan's "skipped-as-success gate via if: always()" requires the workflow to always run first. Fix shipped: the path filter is now an inner dorny/paths-filter job; the gate job always runs and reports.

What worked — headline findings

Force-push and merge-commit rejection

Attempting git push --force-with-lease origin main after the ruleset was applied:

remote: - Cannot force-push to this branch
remote: - Changes must be made through a pull request.
remote: - 3 of 3 required status checks are expected.
! [remote rejected] main -> main
  (push declined due to repository rule violations)

One push attempt confirms US-1, US-2, and the required-check gate.

Skipped-as-success path-filtered CI

On PR #2 (spike-only), required check rollup under the fixed workflow shape:

prod-ci / changes:  SUCCESS
prod-ci / lint:     SKIPPED   ← no prod files
prod-ci / ci-gate:  SUCCESS   ← skip-as-success ✓
ops-ci  / ops-gate: SUCCESS
pre-commit-fast / pass: SUCCESS
spikes-smoke / smoke: SUCCESS

Required checks turn green via skip-as-success exactly as designed (US-7).

CODEOWNER enforcement under count=1

PR #2 (spike-only) and PR #4 (mixed prod+spike) both show REVIEW_REQUIRED; PR #3 (prod-only) does too. CODEOWNERS engages once the approval count is > 0. Dependabot's PR #1 also shows REVIEW_REQUIRED with no reviewer auto-assigned (only one CODEOWNER), proving the rule fires.

LFS round-trip is clean

A 5 MB clip.mp4 committed locally became a 132-byte pointer in git but a 5,242,880-byte binary in the working tree. CI confirms:

with-lfs:  5,242,880 bytes  "data"
without:        132 bytes  "ASCII text"
            version https://git-lfs.github.com/spec/v1

Fresh clone retrieved the full blob automatically — no pointer leaked.

Pre-commit fast / heavy split

trailing-whitespace fires on a spikes/ file (fast stage runs everywhere). ruff with files: ^prod/ reports "(no files to check)" on an ops/ file and a hard 3-error fail on the same file moved to prod/. Scoping works.

Dependabot wakes up fast

The README budgets 24 h. PR #1 (lodash 4.17.0 → 4.18.1) opened within minutes of pushing .github/dependabot.yml. The spike folder is intentionally absent from the config — got no PR, as designed.

AGENTS.md symlinks survive git

Every CLAUDE.md has a sibling AGENTS.md → CLAUDE.md stored as a real git symlink (mode 120000). Delete one and the local guard fails exit 1; restore it and the guard passes. The hook is working.

Large-file rejection

A 2 MB random blob.bin staged in prod/ is rejected at commit time:

check for added large files .... Failed
prod/mock-app/blob.bin (2048 KB) exceeds 1000 KB.

Lessons for the conjure migration

  1. Drop US-8. GitHub doesn't let anyone approve their own PR. Plan needs a second CODEOWNER on prod paths (a teammate, a bot account configured as a real user, or a bypass actor) before the prod tier ships changes.
  2. Decide US-3 / US-5 explicitly. Pick one of: (a) every PR needs a review — simplest, ship now; (b) add a codeowner-required workflow gate that fires only when CODEOWNERS paths are touched — harder but matches plan intent; (c) drop the “spikes can push direct to main” aesthetic.
  3. Internal paths-filter, not top-level paths:. Required checks plus top-level path filters = stuck PRs. Always run the workflow; gate inside.
  4. Dependabot SLA is minutes, not days. Tighten the migration timeline.
  5. US-20 Step B (live bot test) stays open. Schedule it for migration day, not sandbox.
