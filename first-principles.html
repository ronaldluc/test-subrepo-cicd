Per-path monorepo protection — from first principles

The plan from the bottom up: each protective behaviour we’re after is a stack of primitive mechanisms. This page names every primitive once, shows what it lets us build, and only at the end zooms out to mental model and use cases.

Terminology

git object store
A content-addressed key-value store on disk. Keys are SHA-1 hashes. Values are blobs (file content), trees (directory listings), commits, and tags. Identical content has identical key.
ref
A named mutable pointer to a commit SHA. refs/heads/main, refs/heads/feature/x, etc. A branch is just a ref under refs/heads/.
working tree
The set of files on disk in your checkout. Distinct from the object store. Untracked files live in the working tree without ever entering the object store.
protected ref / ruleset
A server-side policy that constrains how a ref may move. Lives on the GitHub side, not in the repo’s objects. Enforced by GitHub’s push receiver.
commit status
A named (context, state, target_url) tuple attached to a commit SHA by some external system. State is success / failure / pending. A ruleset can require named contexts to be success before a ref can advance to that SHA.
workflow
A YAML file under .github/workflows/ that GitHub Actions executes on certain events. Jobs in a workflow publish commit statuses.
pull request
A GitHub object (not a git object) describing a proposed advance of one ref by N commits. Has its own review state, status-check rollup, and merge button.
CODEOWNERS
A file at .github/CODEOWNERS mapping glob patterns to GitHub user/team handles. Read by GitHub when a PR’s diff is evaluated against rulesets.
R2 bucket / S3-compatible object store
Cloudflare’s object storage. Same API surface as AWS S3, distinct billing (zero egress). For this plan: bucket conjure-ops, scoped API token conjure-rw-no-delete, S3 endpoint URL filled in via R2_OPS_ENDPOINT.
asset manifest
A JSON file under ops/ listing R2 objects by key + sha256. The repo stores the manifest; the binary content stays in R2. Fetcher (scripts/ops-fetch.sh) resolves manifest entries on demand and verifies hashes. Pattern lifted from conjure_tech/scripts/fulfill-orders.py.
no-delete IAM token
An R2 (S3) API token scoped to Object Read & Write but not Admin. DELETE requests against the bucket return 403 Forbidden. This is the single mechanical guard preventing code or an LLM from destroying the asset set.
pre-commit
A local-machine framework (and the name of one specific tool) that runs configured hooks before git commit writes anything. Configured by .pre-commit-config.yaml.
CLAUDE.md / @-include
A markdown file Claude Code loads as context when invoked from a directory. @path/to/file.md in a CLAUDE.md transcludes that file’s content. Resolved by Claude Code at session start.
required status check (vs job)
The ruleset references a context string (e.g. ci-gate). That string must appear on the head commit as a status; it doesn’t care which workflow or job produced it. A required context that never appears is treated as missing; the ref stays blocked.

L0 — What git itself does

Git tracks content, not files. git add writes blobs to the object store; git commit writes a tree pointing at those blobs plus a commit object pointing at the tree. Refs (branches) are then advanced to point at the new commit. None of this is networked — that’s git push, which transfers objects to a remote and asks it to update its refs.

Key invariant: git never modifies untracked files. Pull, checkout, merge, reset — all of them operate on tracked content. Untracked files are kept around unless something explicitly deletes them (git clean, rm, the OS). This is the basis for the spike-folder collision result later.

L1 — Server-side ref protection

Primitive: ruleset on main

Stack: GitHub ruleset object → ref-update receiver

A ruleset is a JSON document that GitHub attaches to one or more refs. When a client tries to push, the receiver evaluates the proposed update against the ruleset and accepts or rejects.

The four primitive rules we use:

  • deletion — reject deletion of the ref.
  • non_fast_forward — reject force-pushes (anything that doesn’t cleanly extend the current commit).
  • required_linear_history — the new commit must have one parent (no merge commits).
  • pull_request — the ref can only advance via a PR merge, not a plain push. Has sub-knobs like required_approving_review_count and require_code_owner_review.

Satisfies: US-1 (force-push rejected), US-2 (linear history).

L2 — Required status checks + workflows

Primitive: ruleset requires named contexts on the head commit

Stack: ruleset → commit status ← workflow job

The ruleset has a required_status_checks rule listing context strings (e.g. ci-gate, ops-gate, pass). The ref-update receiver looks up commit-statuses on the head commit and refuses to advance if any required context isn’t success.

Workflows produce those statuses. A job named ci-gate in a workflow named prod-ci publishes a status with context ci-gate (or prod-ci / ci-gate — GitHub normalises this).

The path-filter trap

If a workflow is gated by on: pull_request: paths: [prod/**] at the top level and a PR touches only spikes, the workflow never starts. Its required-context status is never written. Required-context = missing. Ref stays blocked forever.

Fix: the workflow always starts. The first job (changes) uses dorny/paths-filter to set an output. Downstream jobs read that output and either do real work or short-circuit with if: needs.changes.outputs.prod == 'true'. The final aggregator job uses if: always() and reports success if its dependencies are success OR skipped — the “skipped-as-success” pattern.

Satisfies: US-7 (gate green when no prod changes). Foundational for US-4, US-5, US-6.

L3 — Pull-request objects

Primitive: the pull_request ruleset rule

Stack: ruleset → PR object → merge button

With pull_request in the ruleset, no direct push to main is accepted. Every advance of main must come from a PR’s merge. This is a per-branch rule; it cannot be path-scoped (this is why US-3 was dropped).

Sub-knobs:

  • required_approving_review_count: 0 — how many human approvals before merge. We set this to 0 deliberately.
  • require_code_owner_review: true — the next layer; meaningful only when count is ≥ 0 and CODEOWNERS exists.
  • allowed_merge_methods: [squash, rebase] — what kind of advance is allowed. Excludes merge commits (consistent with required_linear_history).

Satisfies: US-2 (PR required); everything downstream that needs a PR object as the carrier (review, CI, doc-protection workflows).

L4 — CODEOWNERS as a path-scoped review gate

Primitive: glob → user/team mapping read at PR-eval time

Stack: .github/CODEOWNERS + require_code_owner_review: true

The CODEOWNERS file maps glob patterns to GitHub handles. When a PR is opened or updated, GitHub computes its diff against these globs. Every matching path requires an approving review from a designated owner who isn’t the author.

The non-obvious knob combination:

countcodeowner=trueBehaviour
1trueEvery PR needs ≥ 1 approving review, codeowner-or-not. Breaks US-5.
0falseNo review required for anything. Breaks US-4.
0trueNo review needed when the diff doesn’t match any codeowner glob. When it does, that codeowner must approve. Path-scoped enforcement from a single ruleset.

Edge case — the sole-codeowner trick: GitHub disallows self-approval. If the PR author is the only codeowner for every path their diff touches, no valid reviewer exists. GitHub silently treats the rule as satisfied. This is how US-8 works without literal self-approval.

Satisfies: US-4 (prod PR needs codeowner review), US-5 (spike-only PR doesn’t), US-6 (mixed PR governed by prod rules), US-8 (author merges own PR when sole owner of touched paths).

L5 — Pre-commit + large-file rejection

Primitive: client-side hook before git commit

Stack: .pre-commit-config.yaml → pre-commit framework → .git/hooks/pre-commit

Pre-commit runs configured hooks against staged files before the commit is written. Hooks are themselves small programs (whitespace fixers, linters, secret detectors). A hook’s non-zero exit aborts the commit; some hooks also modify files in place.

Two stages:

  • Fast stage, scope all paths: trailing whitespace, end-of-file fixer, YAML parser, large-file check (--maxkb=1000), gitleaks, private-key detector.
  • Heavy stage, scope files: ^prod/: ruff lint + format. Only files under prod/ are passed to these hooks.

The same config is mirrored as a CI workflow (pre-commit-fast.yml) and pinned as a required status check (pass). Local pre-commit is convenience; CI is enforcement.

Satisfies: US-9 (fast stage runs everywhere), US-10 (heavy stage scoped), US-11 (large file rejected at commit).

L6 — Binary assets via R2 (no LFS)

Primitive: out-of-git object store + in-git manifest

Stack: ops/…/manifest.json (in git) → scripts/ops-fetch.sh / ops-push.sh → magic/storage/object_store.py → R2 bucket conjure-ops

Git is not asked to track binary content. Each binary asset (xlsx, glb, mp4, model weights, fixtures) lives in R2, addressed by a stable key. A manifest file under ops/<category>/manifest.json lists every asset by key + sha256. The repo stores only the manifest.

Two scripts wrap the existing object-store abstraction:

  • scripts/ops-push.sh — upload a local file to R2, add it to the manifest, commit the manifest. Never deletes existing entries.
  • scripts/ops-fetch.sh — read the manifest, download each object to a gitignored location, verify sha256. Used in CI and locally.

Why this not LFS: LFS bandwidth meters every clone and CI run; bills accumulate on forks; the deletion path is fragile. R2 has zero egress fees and a more native ACL model. Crucially the CI/agent IAM token has no Delete permission — the mechanical guard is much simpler than “don’t run git lfs prune”.

A daily workflow (r2-no-delete-check.yml) attempts to aws s3 rm a canary key with the CI credentials and asserts a 403. If the token ever drifts to having Delete permission, this canary fires.

Satisfies: US-23 (manifest entry resolves to an existing object), US-24 (CI token cannot delete). US-12–US-14 (LFS-flavoured) are retired in this plan.

L7 — Agent context (CLAUDE.md hierarchy)

Primitive: directory-scoped markdown loaded by Claude Code at session start

Stack: CLAUDE.md at the working-directory + @-include resolution + parent-directory walking

When Claude Code starts in a directory, it reads CLAUDE.md at that location and at every parent up to the repo root. Each file may contain @path lines that transclude another markdown file into the loaded context.

The repo organises CLAUDE.mds so each tier loads only what it needs:

  • prod/conjure_tech/frontend/CLAUDE.md @-includes docs/architecture/frontend.md.
  • prod/conjure_tech/backend/CLAUDE.md @-includes docs/architecture/backend.md.
  • ops/CLAUDE.md @-includes only docs/company/principles.md.

Sentinel strings (SENTINEL-FRONTEND-7A3B, SENTINEL-BACKEND-9F1C) embedded in the architecture docs are how we test scope isolation: a frontend session should grep one but not the other.

Satisfies: US-15, US-16 (scope isolation), US-17 (AGENTS.md alias as symlink), US-18 (local hook enforces the symlink convention).

L8 — Bots that look like users

Primitive: GitHub account identity, users/<login>.type

Stack: PR review submitter identity → ai-cannot-approve workflow → required status check

GitHub exposes every account’s type field: User or Bot. github-actions[bot] is type Bot. A dedicated bot account (in this plan, conjure-ops at orgs/conjurehq/people/conjure-ops) is type User.

The ai-cannot-approve.yml workflow listens to pull_request_review events, fetches the reviewer’s type, and fails if a Bot-type account submitted an approving review. This prevents github-actions[bot] — or any other Bot identity — from satisfying CODEOWNER review.

The AI review pipeline (CodeRabbit + Claude Code bot + Chrome MCP) lives on top of this:

  • CodeRabbit posts inline comments and a summary — never an approval.
  • Claude Code via the conjure-ops account uses a User-type identity (so its comments don’t trip the ai-cannot-approve guard), but is deliberately not in CODEOWNERS (so its approvals can’t satisfy required-review). The bot's PAT is the GH_AGENT_PAT repo secret.
  • Chrome MCP runs Claude Code in a headless Chrome to drive UI journeys; emits PR comments with findings.

Satisfies: US-20 (bot approval can’t pass CODEOWNER review); foundational for US-28–30 (the AI review pipeline).

L9 — Docs as load-bearing content

Primitive: workflow that runs a model over the diff

Stack: pull_request event → workflow checks out PR → runs Claude Code with diff + docs as input → publishes status

Docs aren’t throwaway content — they’re what every agent loads as scope. Two workflows protect them:

  • docs-sanity — cheap, fast Haiku call: “did this PR remove docs accidentally?” Threshold-driven. Required check.
  • docs-drift — dispatcher-pattern Claude Code call: read the diff, decide which docs sections plausibly describe the changed code, fan out subagents to compare each, post a PR comment with drift findings. Advisory at first.

Both follow the L2 pattern — workflow always runs, even when docs aren’t touched, so the required-check name always reports.

Satisfies: US-21 (sanity check on accidental deletion), US-22 (drift detection between code and docs).

Mental model

One sentence: main is a guarded ref whose advance is conditioned on (a) shape rules at the git level, (b) contract rules from named status checks, and (c) identity rules from CODEOWNERS-resolved reviews. Every other piece in the plan is either a producer of one of those signals or a layer that runs before the server ever sees a push.

The three tiers, by where they run:

WhereWhenPieces
Developer’s machinepre-pushpre-commit hooks, LFS clean filter, .gitattributes
GitHub serverat ref updateruleset rules (shape + status + PR + codeowner)
GitHub Actions / external botsasynchronously on PR eventsworkflows producing statuses, CodeRabbit, Claude Code bot, Chrome MCP

The dependency direction is one-way: the server-side ruleset cannot pull data from the pre-commit hooks (different machine), but it can require that CI re-runs the same checks. Local hooks are convenience; the only thing that gates the ref is the ruleset.

The four mechanisms that do most of the work

  1. Path-scoped CODEOWNERS + count=0 — gives per-tier review enforcement out of one global ruleset.
  2. Inner paths-filter + if: always() gate job — lets path-scoped CI coexist with required-status-check enforcement.
  3. In-repo manifest + R2 with a no-Delete IAM token — keeps binary content out of git entirely; resolved on demand by scripts/ops-fetch.sh; protected from accidental destruction by IAM-level scope, not git tricks.
  4. CLAUDE.md + @-include — declares per-directory agent scope; loads the right docs without code knowing about it.

Real-world application — one day in the life of a PR

Imagine Alice opens a PR touching prod/conjure_tech/frontend/Button.tsx and spikes/alice/experiment/notes.md.

  1. Locally: she runs git commit. The pre-commit fast stage runs (trailing whitespace, gitleaks, large-file check) on both files. The heavy stage (ruff) runs on the prod file only because its files: ^prod/ rule matches. If a 3 MB asset slipped in, the large-file check rejects it — she has to either trim it, add an LFS pattern, or move it to R2.
  2. She pushes to a feature branch. The ruleset has no opinion about feature branches; it only protects main. Push succeeds.
  3. She opens a PR. GitHub starts every workflow that listens for pull_request. The CI workflows always run; their inner changes jobs say both prod and spike paths changed. prod-ci runs its real lint job. ops-ci’s lint is skipped. The aggregator jobs (ci-gate, ops-gate, pass) all return success, satisfying the required-context list.
  4. CODEOWNERS evaluation: the diff includes a file under prod/**. The ruleset’s require_code_owner_review rule fires. Alice is the author so she can’t approve her own PR. If she’s the sole codeowner of prod/**, GitHub silently treats the rule as satisfied. If a teammate is also listed, that teammate must approve.
  5. AI review: CodeRabbit posts inline style comments within a minute. The Claude-deep-review bot account posts one summary comment with architecture-level observations. Neither submits an approval, so the ai-cannot-approve guard doesn’t fire.
  6. Docs workflows: docs-sanity runs (no docs deleted, passes). docs-drift fans out subagents over docs/architecture/frontend.md and posts a single comment saying which paragraphs may no longer describe the code.
  7. Alice clicks Merge in the web UI. (Or runs gh pr merge — same endpoint.) All required contexts are success; the PR review state satisfies CODEOWNERS or the sole-owner no-op fires. The ref advances. Squash merge produces one commit on main; linear history preserved.

Common misconceptions

No. “Branch protection is path-scoped”. Reality: rulesets attach to refs, not paths. Path scoping comes from CODEOWNERS and from path-aware workflows, both layered on top.
No. “CODEOWNERS enforces review automatically”. Reality: the file is just metadata until the ruleset wires require_code_owner_review: true.
No. “Required status checks need paths: filters in the workflow trigger”. Reality: the opposite. Top-level paths: blocks the workflow from starting on non-matching paths, so its status never gets written. The required context shows as “expected/missing” and the PR is stuck. Filter inside the workflow, not at the trigger.
No. “The author can self-approve their own PR with the right setting”. Reality: GitHub rejects this at the GraphQL layer: “Can not approve your own pull request.” The workaround is the sole-codeowner no-op, or a peer codeowner.
No. “We use Git LFS for binaries”. Reality: we don’t. The plan uses R2 + an in-repo manifest. LFS bandwidth meters every clone and CI run, bills accumulate on forks, and the deletion-protection story is messy. R2 has zero egress fees, and the IAM no-Delete token is a simpler and stronger guarantee than anything LFS offers.
No. “git pull can lose my untracked spike files when someone deletes the folder”. Reality: git never touches untracked files during pull. The folder is preserved if untracked content remains. The only command that destroys untracked files is git clean -fd.
No. “UI-only merge can be enforced by GitHub config”. Reality: the web UI and gh pr merge hit the same REST endpoint. No native distinction. This is a policy/cultural choice, not a config.
No. “Workflows skip implicitly when their paths aren’t touched, that’s good”. Reality: implicit skip via top-level paths: means no status is published, which makes required checks block forever. Explicit skip-as-success via an always-running aggregator job is what works.

Summary — five bullets

  1. main is a guarded ref. Server-side rules (no force-push, linear history, required PR, required status checks) describe what shape and provenance of update is acceptable.
  2. Required status checks bind ref advance to external contracts, fulfilled by workflow jobs. The workflows must always start and gate work internally with dorny/paths-filter + if: always() aggregator — otherwise required contexts go missing.
  3. CODEOWNERS plus required_approving_review_count = 0 plus require_code_owner_review = true is what gives path-scoped review enforcement out of one ruleset, including the sole-codeowner no-op that lets the author merge their own prod PR.
  4. Pre-commit hooks reject > 1 MB files at commit; binary content lives in R2 (bucket conjure-ops) addressed by an in-repo manifest. The CI/agent IAM token has no Delete, enforced by a daily canary workflow.
  5. Agent context (nested CLAUDE.md + sentinels) and AI review (CodeRabbit + Claude Code bot User-account + Chrome MCP) layer on top of these primitives; bot-as-User identity plus the ai-cannot-approve workflow stop automation from gaming the review system.
