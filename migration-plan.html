Phase 4 — R2 asset storage (no LFS)

~60 min · reuses existing conjure tooling

Decision recap

No git LFS at all. Every binary asset (xlsx, glb, mp4, model files, fixtures > 1 MB) lives in Cloudflare R2. The git repo stores only hash-pinned references in ops/orders/manifest.json (and similar manifest files per use case). This is the same pattern conjure_tech/scripts/fulfill-orders.py already uses for STL downloads.

4.1 — Reuse the existing object-store abstraction

Conjure already ships a Python S3-compatible client at magic/storage/object_store.py (Protocol + MinIO + Mock implementations). When you import conjure into prod/conjure_tech/ in Phase 2, the file moves to prod/conjure_tech/magic/storage/object_store.py. The repo-root tooling we’re about to write delegates to that module rather than duplicating S3-client code.

4.2 — Create the conjure-ops R2 bucket

You can re-run the existing bootstrap (port from conjure_tech/scripts/setup-r2.sh) with a different bucket name, or do it once in the UI:

UI Cloudflare dashboard → R2 → Create bucket. Name: conjure-ops. Same account as conjure-artifacts. Direct URL: https://dash.cloudflare.com → R2.

Keep conjure-artifacts as-is — it serves the live API. conjure-ops is the new bucket for repo-tracked assets (orders xlsx, design files, model weights, fixtures).

4.3 — Mint the scoped no-delete token most important step

UI R2 → Manage API Tokens → Create API Token:

  • Name: conjure-rw-no-delete
  • Permissions: Object Read & Write (not Admin)
  • Buckets: conjure-ops only (do not multi-select with conjure-artifacts)
  • TTL: 1 year

Save Access Key, Secret Key, and the account-scoped S3 endpoint URL (looks like https://<account-hash>.r2.cloudflarestorage.com) to 1Password under Conjure / Dev Secrets → r2-conjure-ops. Add to GitHub repo secrets as R2_OPS_ACCESS_KEY, R2_OPS_SECRET_KEY, R2_OPS_ENDPOINT.

Critical check — do this immediately: from a shell with those credentials, attempt a delete: 
AWS_ACCESS_KEY_ID=<key> AWS_SECRET_ACCESS_KEY=<secret> \
  aws s3 rm s3://conjure-ops/test-object --endpoint-url=<R2_OPS_ENDPOINT>
Must return 403 Forbidden. If it succeeds, the token has Delete permission — re-mint with the correct scope before anything else uses it. This is the single guard preventing an LLM or a runaway script from destroying the asset set.

4.4 — The manifest pattern

One manifest per "set of related assets" lives under ops/. Example for the orders tracker:

cat > ops/orders/manifest.json <<'EOF'
{
  "$schema": "../../scripts/asset-manifest.schema.json",
  "bucket": "conjure-ops",
  "objects": [
    {
      "key": "orders/2026-Q2.xlsx",
      "sha256": "<REPLACE_WITH_HASH>",
      "size_bytes": 0,
      "uploaded_at": "2026-05-15",
      "description": "Q2 2026 testing-order tracker"
    }
  ]
}
EOF

What lives in git: this manifest + small docs + scripts. What lives in R2: the actual .xlsx / .glb / .safetensors bytes. The repo never grows because of binary content.

4.5 — The fetcher/pusher

Two thin scripts at the repo root, both wrapping prod/conjure_tech/magic/storage/object_store.py:

scripts/ops-fetch.sh <manifest.json>
  # Reads manifest entries, downloads each to ./ops/<path>/<key-basename>
  # Verifies sha256. Fails if any object missing or hash mismatch.
  # Uses R2_OPS_* env vars in CI; locally sources from 1Password CLI (`op read`).

scripts/ops-push.sh <local-file> <manifest.json>
  # Uploads <local-file> to R2 with key derived from filename + content hash.
  # Updates the manifest with the new entry. Never deletes existing entries.
  # Will fail loudly if the R2 token has Delete permission (canary check).

Add the manifest paths to .gitignore’s asset patterns so resolved files aren’t accidentally re-committed:

# .gitignore
ops/**/resolved/
ops/**/*.xlsx
ops/**/*.glb
ops/**/*.safetensors
ops/**/*.mp4

Phase 13 — Per-product make test + root dispatcher

The root Makefile is a dispatcher; each product owns its real targets.

cat > Makefile <<'EOF'
# Root dispatcher. Args:
#   PATHS=   (CI passes this)
#   PRODUCT=                          (manual override)

PATHS ?=
PRODUCT ?=

.PHONY: test lint typecheck

test:
ifdef PRODUCT
	$(MAKE) -C prod/$(PRODUCT) test
else
	@for p in $$(./scripts/products-from-paths.sh $(PATHS)); do \

	done

lint:
ifdef PRODUCT
	$(MAKE) -C prod/$(PRODUCT) lint
else
	@for p in $$(./scripts/products-from-paths.sh $(PATHS)); do \
	  $(MAKE) -C prod/$$p lint; \
	done
endif

typecheck: ## same shape as test/lint
EOF

Each prod/<product>/Makefile defines real targets. For conjure_tech, port the existing Makefile targets from the old conjure repo.

Failure modes and rollback

If …Then …
filter-repo produces a broken historythrow away conjure-import/ and start Phase 2 again. The new repo on GitHub stays empty until you push. Idempotent.
Ruleset blocks every PR (required checks misnamed)delete the ruleset (gh api -X DELETE /repos/…/rulesets/<id>); fix the contexts; re-create. You retain admin so nothing locks you out.
R2 token has Delete permission by mistakerevoke at the Cloudflare dashboard; re-mint with correct scope; update repo secrets.
CodeRabbit floods PRs with noisetune .coderabbit.yaml path filters; if unsalvageable, uninstall the App and stick with Claude-deep-review only.
claude-deep-review costs spikelower --max-cost-usd in the workflow; set a daily org-wide Anthropic spending cap; disable on non-prod PRs.
Production deploy regresses after Phase 15.2roll back the deploy via Netlify/Wrangler/Railway; the new repo stays as-is, old conjure repo isn’t archived yet so it can resume serving.

Open questions that need human decisions before starting

  1. Do we want a second human reviewer on prod/ from day one, or accept solo-codeowner no-op for the first weeks? If second reviewer is added, every prod PR needs their approval — slower but safer.
  2. Is conjurehq/conjure the right name, or should we name it conjurehq/conjure-v2 / conjurehq/platform? Renaming later is cheap but breaks bookmarks.
  3. Migrate open PRs and issues from conjurehq/conjure? GitHub has no native transfer; need a script (gh issue list | xargs gh issue create). Decide whether the cost is worth it for outstanding work.
  4. Repository visibility: stay private (default) or open-source after migration? If open-sourcing, LFS bandwidth on forks becomes a real cost — switch to R2 for large assets first.
